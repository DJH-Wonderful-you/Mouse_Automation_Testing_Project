## 鼠标项目

下面是这次修复的完整流程记录。

**1. 第一轮定位：为什么“已连接鼠标检测不到”**

1. 检查内容：先查蓝牙检测代码和日志来源，定位到 [bluetooth_probe.py:64](https://file+.vscode-resource.vscode-cdn.net/d%3A/RJHZ/Vscode-extensions/extensions/openai.chatgpt-26.304.20706-win32-x64/webview/) 的 Get-PnpDevice 过滤条件。
   原因：日志里只出现耳机，不出现鼠标，明显像被过滤掉。
   结果：代码只允许 BTHENUM|BTHLEDEVICE。
2. 检查内容：在系统侧枚举真实设备实例 ID。
   原因：确认鼠标在 Windows 里到底是什么前缀。
   结果：你的鼠标是 BTHLE\DEV_...，不在原过滤范围内。
3. 结论：首个根因是“前缀漏判”。
   修正：把 BTHLE 纳入识别，并补单测。

**2. 第二轮定位：为什么鼠标关电后仍显示已连接**

1. 检查内容：审查连接判定逻辑 any(device.present and device.status_ok)，位置在 [bluetooth_probe.py:126](https://file+.vscode-resource.vscode-cdn.net/d%3A/RJHZ/Vscode-extensions/extensions/openai.chatgpt-26.304.20706-win32-x64/webview/)。
   原因：这个逻辑会把“已配对且驱动正常”误当成“当前在线”。
   结果：逻辑确实过于粗。
2. 检查内容：读取 Get-PnpDeviceProperty（IsPresent/Status/DevNodeStatus/...）。
   原因：找“当前连接态”字段。
   结果：BLE 鼠标经常在关电后仍是 Present=True + Status=OK，且有 AlwaysShowDeviceAsConnected=True，不能直接用。
3. 检查内容：尝试 AEP IsConnected（WinRT/PnP 属性）。
   原因：这是理论上的实时连接字段。
   结果：本机该路径返回空值，不可用。
4. 结论：需要改成“BLE 主设备 + HID 服务节点联动判定”。

**3. 第三轮定位：修完后又变成“总是未连接”**

1. 检查内容：打印内部映射（MAC->HID签名、HID在线签名集）。
   原因：确认是映射失败还是在线集合为空。
   结果：映射有值，但 hid_connected_signatures 为空。
2. 检查内容：分组统计脚本抓到的节点类型（BT/HID）。
   原因：验证是否真的抓到了 HID 节点。
   结果：当时只抓到 BT，HID=0。原因是 PowerShell 正则转义写错。
3. 结论：回归根因是 HID 查询正则错误，导致新判定链失效。

**4. 最终修正内容**

1. 蓝牙检测脚本修正并保留回退策略：
   位置 [bluetooth_probe.py:68](https://file+.vscode-resource.vscode-cdn.net/d%3A/RJHZ/Vscode-extensions/extensions/openai.chatgpt-26.304.20706-win32-x64/webview/), [bluetooth_probe.py:233](https://file+.vscode-resource.vscode-cdn.net/d%3A/RJHZ/Vscode-extensions/extensions/openai.chatgpt-26.304.20706-win32-x64/webview/)。
   做法：
   - 正确抓取 HID\{00001812...}。
   - 只有在“该 MAC 的 HID 签名确实存在”时才用 HID 判定；否则回退到基础 present+status。
     目的：既提升准确性，又避免某些机器上抓不到 HID 时误判全离线。
2. 查询超时从 8s 提到 15s：
   位置 [bluetooth_probe.py:305](https://file+.vscode-resource.vscode-cdn.net/d%3A/RJHZ/Vscode-extensions/extensions/openai.chatgpt-26.304.20706-win32-x64/webview/)。
   原因：加入 HID 后枚举更慢，避免偶发超时导致空结果。
3. 新增手动检查控件：
   按钮位置 [power_cycle_tab.py:305](https://file+.vscode-resource.vscode-cdn.net/d%3A/RJHZ/Vscode-extensions/extensions/openai.chatgpt-26.304.20706-win32-x64/webview/)。
   逻辑位置 [power_cycle_tab.py:568](https://file+.vscode-resource.vscode-cdn.net/d%3A/RJHZ/Vscode-extensions/extensions/openai.chatgpt-26.304.20706-win32-x64/webview/)。
   启用状态管理 [power_cycle_tab.py:852](https://file+.vscode-resource.vscode-cdn.net/d%3A/RJHZ/Vscode-extensions/extensions/openai.chatgpt-26.304.20706-win32-x64/webview/)。
   功能：输入名称/MAC 后可立即校验并输出匹配设备与连接判定。
4. 单测补强：
   位置 [test_bluetooth_matcher.py:34](https://file+.vscode-resource.vscode-cdn.net/d%3A/RJHZ/Vscode-extensions/extensions/openai.chatgpt-26.304.20706-win32-x64/webview/)。
   覆盖 BLE-HID 映射、签名提取、回退逻辑。

**5. 验证结果**

1. 全量测试：python -m unittest discover -s tests -p "test_*.py" -v，24/24 通过。
2. 实机查询：query_bluetooth_devices() 已能给出鼠标 connected=True/False，不再固定全“已连接”或全“未连接”。
3. is_target_connected('Logi M750', '', 'name_or_mac') 返回 True（在当时在线状态下）。

**为什么采用这套检测方式**

1. Get-PnpDevice 是当前项目已用、权限和兼容性最稳定的通道。
2. 仅用 Status/Present 对 BLE 鼠标不可靠。
3. 用 BTHLEDevice(1812) 与 HID(1812)联动，能更接近“输入设备是否在线”这一业务目标。
4. 加回退是为了跨机器鲁棒性，避免因某台机抓不到 HID 节点而全量误判。
