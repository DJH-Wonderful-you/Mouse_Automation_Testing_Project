**1. 第一轮定位：为什么“已连接鼠标检测不到”**

1. 检查内容：先查蓝牙检测代码和日志来源，定位到 [bluetooth_probe.py:64](https://file+.vscode-resource.vscode-cdn.net/d%3A/RJHZ/Vscode-extensions/extensions/openai.chatgpt-26.304.20706-win32-x64/webview/) 的 Get-PnpDevice 过滤条件。
   原因：日志里只出现耳机，不出现鼠标，明显像被过滤掉。
   结果：代码只允许 BTHENUM|BTHLEDEVICE。
2. 检查内容：在系统侧枚举真实设备实例 ID。
   原因：确认鼠标在 Windows 里到底是什么前缀。
   结果：你的鼠标是 BTHLE\DEV_...，不在原过滤范围内。
3. 结论：首个根因是“前缀漏判”。
   修正：把 BTHLE 纳入识别，并补单测。

**2. 第二轮定位：为什么鼠标关电后仍显示已连接**

   结果：逻辑确实过于粗。

   原因：找“当前连接态”字段。
   结果：BLE 鼠标经常在关电后仍是 Present=True + Status=OK，且有 AlwaysShowDeviceAsConnected=True，不能直接用。

   原因：这是理论上的实时连接字段。
   结果：本机该路径返回空值，不可用。
4. 结论：需要改成“BLE 主设备 + HID 服务节点联动判定”。

也就是说，手工判断时：

- `Started` 等价于已连接
- `OK` 等价于已连接
- `Disconnected`、`Unknown`、`Not Present` 等价于未连接

**快速检测和回退**

为了加快“连续点检测连接状态”的速度，代码会缓存 watcher，逻辑在 [bluetooth_probe.py](d:\RJHZ\Project_Code\Python\Mouse_Automation_Testing_Project\src\core\bluetooth_probe.py#L140) 和 [bluetooth_probe.py](d:\RJHZ\Project_Code\Python\Mouse_Automation_Testing_Project\src\core\bluetooth_probe.py#L697)。

当前行为是：

- 首次检测时，先全量枚举，找到目标设备和对应 watcher
- 后续短时间内再次检测时，不再全量扫，而是直接：
  - `pnputil /enum-devices /instanceid "<watcher_id>"`

watcher 的选择顺序在 [bluetooth_probe.py](d:\RJHZ\Project_Code\Python\Mouse_Automation_Testing_Project\src\core\bluetooth_probe.py#L697)：

1. 优先 `HID\{00001812...}` 实例
2. 如果没有，再用 `BTHLEDevice\{00001812...}` 服务实例
3. 再不行，才用主蓝牙节点自己的实例 ID

这就是我前面补的 Win10 兼容点。因为有些 Win10 机器没有独立 `HID\...` 行，但有 `BTHLEDevice\{00001812...}` 行。如果只看主蓝牙节点，可能一直是 `Disconnected`；看服务节点反而能得到 `OK`。

**你可以手工执行的命令**

先按这个顺序查，最有效。

1. 看是否已配对存在

```powershell
pnputil /enum-devices /class Bluetooth
```

判断方法：

- 有 `LOWA Mouse` 这样的设备名，且对应实例像 `BTHLE\Dev_xxx` 或 `BTHENUM\Dev_xxx`，说明设备存在于已配对列表
- 没有目标设备名，说明“已配对检测”本身就过不去

2. 看是否有 HID 服务节点

```powershell
pnputil /enum-devices /class HIDClass
```

重点找这两类：

```text
HID\{00001812-0000-1000-8000-00805F9B34FB}_DEV_...
BTHLEDevice\{00001812-0000-1000-8000-00805F9B34FB}_Dev_...
```

判断方法：

- 如果找到对应设备的 `00001812` 节点，并且 `Status` 是 `OK` 或 `Started`，基本可判定已连接
- 如果只有 `BTHLEDevice\{00001812...}` 是 `OK`，而主节点 `BTHLE\Dev_xxx` 是 `Disconnected`，当前代码也会把它判成已连接
- 如果两者都 `Disconnected`，通常就是未连接

3. 精确看某一个节点

```powershell
pnputil /enum-devices /instanceid "完整实例ID"
```

这一步适合你从前两步里复制某一行的 `Instance ID` 进去。

4. 如果 `pnputil /format csv` 在那台机子不支持，直接用这些替代

```powershell
pnputil /enum-devices /class Bluetooth /format xml
pnputil /enum-devices /class HIDClass /format xml
```

或者最稳妥：

```powershell
pnputil /enum-devices /class Bluetooth
pnputil /enum-devices /class HIDClass
```

5. 如果 `pnputil` 全部不靠谱，再用 PowerShell 兜底

```powershell
Get-PnpDevice | Where-Object {
    $_.InstanceId -match '^(BTHENUM|BTHLEDEVICE|BTHLE)\\' -or
    $_.InstanceId -match '^HID\\\{00001812-0000-1000-8000-00805F9B34FB\}_DEV_'
} | Select-Object Status, Class, FriendlyName, InstanceId, Present
```

**你拿到结果后怎么直接判断**

你可以按这个简化规则自己判：

1. `Bluetooth` 类里存在目标主节点
   说明：设备已配对过

2. `HIDClass` 里存在对应的 `HID\{00001812...}` 节点且状态是 `OK/Started`
   说明：已连接

3. 如果没有上面的 `HID\...`，但有 `BTHLEDevice\{00001812...}` 节点且状态是 `OK/Started`

   说明：只能看它自己的状态
   `OK/Started` 算连接，`Disconnected` 算未连接

5. 如果 `Present=False`，不管名字多像，按当前逻辑都算未连接
